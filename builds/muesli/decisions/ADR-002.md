# ADR-002: Tokenized array fields for full-text search instead of external search engine

## Context

Q-002 flagged Firestore's limited full-text search. Options: Elasticsearch/Typesense as external dependency, Algolia as SaaS, or Firestore-native approach with tokenized arrays.


## Decision

Store lowercase tokenized words as searchTokens array fields on meetings, segments, actions. Use Firestore array-contains-any queries (max 10 disjunctions) for search. Tokenization: lowercase, split on whitespace/punctuation, deduplicate, filter stopwords. For multi-word queries, intersect results client-side from parallel array-contains queries.


## Consequences

Pro: No external dependency, zero additional cost, works within Firestore's model. Con: No fuzzy matching, no relevance ranking (must score client-side), limited to 10 tokens per query, no phrase matching. Adequate for single-tenant meeting search where the primary discovery path is semantic search (ADR-001) and this serves as a fallback for exact keyword lookup. Can add Typesense later if search quality becomes a pain point.
