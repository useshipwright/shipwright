# ADR-005: Fastify plugin encapsulation for cross-cutting concerns

## Context

The service has 5 cross-cutting concerns (auth, rate limiting, request context, audit logging,
error handling) that must compose correctly with 22 route endpoints. Need a registration order
that ensures auth runs before rate limiting, rate limiting before handlers, and audit logging is
available to all mutation handlers.


## Decision

Each cross-cutting concern is a Fastify plugin registered in strict order:
1. log-redactor (logger config at Fastify construction)
2. request-context (onRequest — earliest, generates requestId)
3. metrics (onRequest/onResponse — wraps all requests)
4. api-key-auth (onRequest — after context, before everything else)
5. rate-limiter (preHandler — after auth, body parsed for key identification)
6. error-handler (setErrorHandler — catches all)
7. audit-logger (decorator — called explicitly by handlers, not a hook)
Routes registered last, after all plugins.


## Consequences

+ Deterministic execution order — no subtle hook ordering bugs.
+ Each plugin is independently testable via Fastify's encapsulation.
+ Audit logging is explicit (handler calls emitAudit) rather than implicit — avoids audit entries
  for failed operations.
- Plugin registration order is load-bearing — must be documented and enforced in server.ts.
