# ADR-012: /metrics endpoint intentionally unauthenticated

## Context

The /metrics endpoint serves Prometheus-format telemetry for scraping by monitoring infrastructure.
Prometheus scrapers expect unauthenticated HTTP GET access. Adding API key authentication to /metrics
would require configuring the scraper with a key, complicating monitoring setup and creating a
circular dependency (monitoring depends on a secret it should be monitoring the availability of).

The threat model identifies "Information leakage via /metrics endpoint" as a medium-likelihood,
medium-impact risk: path-labelled counters reveal which endpoints are used, firebase_admin_calls_total
reveals SDK usage patterns, and rate_limit_exceeded_total reveals attack attempts.

## Decision

/metrics (and /health) remain exempt from API key authentication. This is an intentional trade-off:
operational observability over information hiding. The risk is acceptable because Cloud Run
deployments use internal-only ingress by default — only services within the same VPC (or with
Cloud Run invoker permissions) can reach the endpoint.

Operators MUST ensure one of the following access controls is in place:

1. **Cloud Run ingress = internal-only** (default for pack deploy) — restricts all traffic to
   VPC-internal sources.
2. **Network policy / VPC firewall rules** — if ingress is relaxed for other reasons, restrict
   /metrics access at the network layer.
3. **Cloud Run IAM invoker binding** — limit which service accounts can invoke the Cloud Run
   service at all.

## Consequences

+ Standard Prometheus scraping works without special auth configuration.
+ No circular dependency between monitoring and secret management.
+ Cloud Run internal-only ingress provides defence-in-depth.
- An attacker with VPC access can read metrics without an API key.
- Metrics labels reveal endpoint usage patterns (mitigated by VPC boundary).
- Operators who relax ingress settings must add compensating controls.
