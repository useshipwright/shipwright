# ADR-022: Custom IAM Roles with Narrower Permissions

## Context

The threat model identifies "Privilege Escalation via Operator Over-Provisioning" as a medium-likelihood, high-impact threat. Mitigation (3) specifies: "consider custom roles with narrower permissions instead of predefined roles."

The RBAC matrix currently assigns 6 predefined roles to the operator:

| Predefined Role | Blast Radius |
|----------------|-------------|
| `roles/run.admin` | Full control over ALL Cloud Run services in the project |
| `roles/cloudbuild.builds.editor` | Manage ALL builds in the project |
| `roles/secretmanager.admin` | Full access to ALL secrets in the project |
| `roles/artifactregistry.admin` | Full access to ALL repositories in the project |
| `roles/iam.serviceAccountUser` | Can act as ANY service account in the project |
| `roles/resourcemanager.projectIamAdmin` | Can modify ANY IAM binding in the project |

The Cloud Build SA has 3 predefined roles (`roles/artifactregistry.writer`, `roles/run.developer`, `roles/iam.serviceAccountUser`), each granting broader access than needed.

A compromised operator account with these predefined roles could escalate privileges across the entire GCP project — not just the firebase-auth service resources. Similarly, a compromised Cloud Build SA could deploy to any Cloud Run service and push to any Artifact Registry repository.

## Decision

Define two project-level custom IAM roles containing only the exact permissions required by the firebase-auth service:

1. **`firebaseAuthPackOperator`** — Replaces 6 predefined operator roles with a single custom role containing ~35 scoped permissions.

2. **`firebaseAuthPackBuilder`** — Replaces 3 predefined Cloud Build SA roles with a single custom role containing ~10 scoped permissions.

Custom roles are defined in `template/iam/custom-roles.yaml` and managed by `scripts/deploy.sh`.

### Why custom roles reduce risk

| Property | Predefined Roles | Custom Roles |
|----------|-----------------|-------------|
| Scope | All resources of that type project-wide | Only permissions needed for this service |
| Audit surface | 6 separate role bindings to track | 1 role definition to audit |
| Privilege creep | Predefined roles gain new permissions over time | Custom roles are frozen at creation |
| Blast radius (operator) | Can manage ALL Cloud Run services, secrets, repos, IAM | Can only manage service-specific resources |
| Blast radius (build SA) | Can push to ANY repo, deploy to ANY service | Can only push/deploy for this service |

### Key permission reductions

**Operator — removed permissions (compared to predefined):**
- `secretmanager.versions.access` — Operator can manage secrets but NOT read secret values. This prevents the operator from exfiltrating the Firebase SA credential.
- `cloudbuild.builds.update` — Operator can submit and view builds but not modify in-flight builds.
- `artifactregistry.tags.*` — Operator can manage repos but tag management is not needed.
- `run.jobs.*` — Only Cloud Run services are used, not jobs.
- All `*.delete` permissions on resources outside this service's scope.

**Cloud Build SA — removed permissions (compared to predefined):**
- `run.services.create` / `run.services.delete` — Build SA can update existing services but NOT create or delete them.
- `artifactregistry.repositories.create` / `*.delete` — Build SA can push images but NOT manage repositories.

### Runtime SA — no change

The runtime service account (`firebase-auth-sa`) already uses scoped predefined roles:
- `roles/secretmanager.secretAccessor` bound per-secret (ADR-017)
- `roles/logging.logWriter` (write-only, low risk)
- `roles/cloudtrace.agent` (write-only, low risk)

These are already minimally scoped. Custom roles for the runtime SA would add complexity without meaningful security benefit.

### Deployment

The deploy script (`scripts/deploy.sh`) manages custom roles in a new Phase 0f:

1. Parse role definitions from `iam/custom-roles.yaml`
2. Check if each custom role exists in the project
3. Create the role if missing, or update if permissions have changed
4. Bind the custom role to the appropriate principal

The `SKIP_CUSTOM_ROLES=true` environment variable allows falling back to predefined roles for environments where custom role creation is not permitted (e.g., projects with org policy constraints on custom roles).

### Limitations

- **Project-level scope**: GCP custom roles are project-level, not resource-level. A principal with `firebaseAuthPackOperator` can still use the included permissions on any matching resource in the project. True resource-level scoping requires IAM Conditions (deferred).
- **Permission availability**: Some predefined role permissions cannot be included in custom roles. The definitions in `iam/custom-roles.yaml` include only testable, supported permissions.
- **Org policy constraints**: Some organizations restrict custom role creation. The `SKIP_CUSTOM_ROLES` escape hatch addresses this.

## Consequences

- Operator and Cloud Build SA permissions are documented, auditable, and frozen in version control
- A compromised operator cannot read secret values (`secretmanager.versions.access` excluded)
- A compromised Cloud Build SA cannot create/delete Cloud Run services or manage repos
- Custom roles must be updated explicitly when new permissions are needed (intentional friction)
- The `SKIP_CUSTOM_ROLES` flag provides a fallback path for constrained environments
- Role definitions are versioned alongside the service — changes go through code review
