# ADR-018: Enable DATA_READ Audit Logs for Secret Manager

## Context

The threat model identifies "Service Account Key Exfiltration" as a high-impact risk. Mitigation (2) specifies "Secret Manager audit logging" to detect unauthorized secret access. GCP Data Access audit logs are not enabled by default — they must be explicitly turned on per service.

Without DATA_READ audit logs on `secretmanager.googleapis.com`, there is no record of which principals accessed the `FIREBASE_SERVICE_ACCOUNT_JSON` secret value. A compromised service account could read the secret without leaving a trace in Cloud Audit Logs.

## Decision

The deploy script (`scripts/deploy.sh`) verifies that DATA_READ audit logs are enabled for `secretmanager.googleapis.com` in the GCP project. If not enabled, the script warns the operator and provides the gcloud command to enable them.

### Why DATA_READ specifically?

| Log Type | What It Captures | Default |
|----------|-----------------|---------|
| ADMIN_READ | Secret metadata (list, describe) | Enabled by default |
| DATA_READ | Secret value access (versions.access) | **Disabled by default** |
| DATA_WRITE | Secret version creation/destruction | Enabled by default |

`DATA_READ` captures `SecretManagerService.AccessSecretVersion` — the exact API call the Cloud Run service makes to read `FIREBASE_SERVICE_ACCOUNT_JSON` at startup. This is the critical event for detecting exfiltration.

### Implementation

1. **Deploy script Phase 0e** fetches the project IAM policy and checks `auditConfigs` for `secretmanager.googleapis.com` with `DATA_READ` in `auditLogConfigs`.

2. If DATA_READ is missing, the script emits a **warning** (non-blocking) with the exact gcloud command to enable it. Auto-enabling is avoided because modifying the project IAM policy requires a read-modify-write cycle that risks concurrent modification conflicts.

3. The operator can enable audit logging with:

```bash
# Get current policy
gcloud projects get-iam-policy "${GCP_PROJECT_ID}" --format=json > /tmp/policy.json

# Add auditConfig for Secret Manager DATA_READ (if not already present)
# Then apply:
gcloud projects set-iam-policy "${GCP_PROJECT_ID}" /tmp/policy.json
```

Or via the GCP Console: IAM & Admin > Audit Logs > Secret Manager API > enable "Data Read".

### Why warn instead of auto-enable?

- `gcloud projects set-iam-policy` is a full policy replacement — concurrent modifications can drop IAM bindings
- Audit log configuration is a project-wide setting affecting all Secret Manager usage, not just this service
- The operator should make this decision explicitly, especially in shared projects

## Consequences

- Deploy script detects missing DATA_READ audit logs and warns the operator
- Once enabled, all `AccessSecretVersion` calls are logged in Cloud Audit Logs
- Security teams can monitor for unexpected secret access patterns
- Slight increase in Cloud Logging costs (DATA_READ logs are billable)
- No runtime impact on the application — audit logging is a GCP platform feature
