# ADR-017: Per-Secret IAM Binding for Secret Manager Access

## Context

The RBAC matrix specifies that the Service Account (runtime) needs `roles/secretmanager.secretAccessor` to read `FIREBASE_SERVICE_ACCOUNT_JSON` from Secret Manager. The threat model identifies "Service Account Key Exfiltration" as a high-impact risk and lists "per-secret IAM binding (not project-wide)" as mitigation (1).

**The risk:** If `roles/secretmanager.secretAccessor` is granted at the project level, the Cloud Run service account can read **every** secret in the GCP project — not just its own credential. A compromised service could exfiltrate database passwords, API keys, or other services' credentials.

**The fix:** Bind the role on the specific secret resource, so the service account can only access `firebase-auth-sa-key` and nothing else.

## Decision

The deploy script (`scripts/deploy.sh`) binds `roles/secretmanager.secretAccessor` at the **secret resource level**, not at the project level. It also audits existing project-level bindings and warns if the service account has broader access than necessary.

### Implementation

1. **IAM binding phase** (Phase 0d in deploy.sh) uses `gcloud secrets add-iam-policy-binding` to grant `roles/secretmanager.secretAccessor` to the Cloud Run service account on the specific secret `firebase-auth-sa-key`.

2. **IAM audit phase** checks whether the service account has project-level `roles/secretmanager.secretAccessor` and warns if so (this would be over-provisioned).

3. The binding is idempotent — running it multiple times does not create duplicate bindings.

### Why per-secret, not project-level?

| Scope | Access | Blast radius |
|-------|--------|-------------|
| Project-level | All secrets in the project | Compromise exposes every secret |
| Per-secret | Only `firebase-auth-sa-key` | Compromise exposes only this service's credential |

Least-privilege principle: the service account should only access exactly the secrets it needs.

### gcloud commands

```bash
# Per-secret binding (correct)
gcloud secrets add-iam-policy-binding firebase-auth-sa-key \
  --project="${GCP_PROJECT_ID}" \
  --member="serviceAccount:${SERVICE_ACCOUNT}" \
  --role="roles/secretmanager.secretAccessor"

# Project-level binding (AVOID — over-provisioned)
gcloud projects add-iam-policy-binding "${GCP_PROJECT_ID}" \
  --member="serviceAccount:${SERVICE_ACCOUNT}" \
  --role="roles/secretmanager.secretAccessor"
```

## Consequences

- The Cloud Run service account can only read `firebase-auth-sa-key`, not other secrets in the project
- Each deployment gets isolated secret access — no cross-service secret leakage
- The deploy script handles the IAM binding automatically during deployment
- Operators are warned if project-level `secretAccessor` bindings exist (over-provisioned)
- Adding new secrets for this service requires adding new per-secret bindings (explicit, auditable)
