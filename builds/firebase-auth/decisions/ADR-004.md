# ADR-004: Fail-fast on invalid Firebase credentials

## Context

Q-012 asks whether the service should fail to start or return 503 when credentials are invalid. initializeApp() does not validate credentials synchronously — cert() only checks structure.


## Decision

Fail-fast strategy. The firebase plugin calls cert() with the parsed JSON — if the structure is invalid (missing projectId, clientEmail, privateKey), cert() throws and the plugin fails, preventing Fastify from starting. If the structure is valid but the key is wrong, the service starts (cert() succeeds) but the first verifyIdToken() with checkRevoked=true or getUser() call will fail. The health endpoint reports firebase_initialized=true for structural validity. A future enhancement could make a probe getUser() call during init.


## Consequences

- Structurally invalid credentials: container fails to start, Cloud Run shows revision error
- Structurally valid but wrong credentials: service starts, basic verifyIdToken works (public key verification only), getUser fails at runtime
- Health endpoint reflects init status, not credential validity — acceptable for v1
