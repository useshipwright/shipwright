# ADR-004: Fail fast on Firebase SDK initialization failure

## Context

Q-012 asks whether the service should start in degraded mode or fail fast when Firebase SDK
initialization fails. The service is entirely dependent on Firebase — every endpoint except /health
requires a working SDK connection.


## Decision

Fail fast. During startup, the firebase-adapter performs a probe call (getUser with a known-nonexistent
UID). If the call fails with anything other than auth/user-not-found, the service throws and does not
call fastify.listen(). Cloud Run's startup probe will detect the failure and restart the container.
The /health endpoint still reports firebase: 'error' for transient runtime failures after successful
startup.


## Consequences

+ Bad credentials detected immediately — no silent failures where all 22 endpoints return 500.
+ Cloud Run restart policy handles recovery (exponential backoff on repeated failures).
+ Consistent with Cloud Run's startup probe pattern (no traffic until probe passes).
- One extra Firebase API call at startup (~100ms). Acceptable tradeoff for deployment confidence.
