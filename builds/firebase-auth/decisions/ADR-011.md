# ADR-011: Test strategy — manual Fastify construction for route tests

## Context

buildApp({ skipFirebaseInit: true }) does not register route plugins (they depend on the firebase plugin). Prior attempts to use app.decorate after buildApp failed because routes are registered at build time, not request time.


## Decision

Health tests use buildApp({ skipFirebaseInit: true }) — health route is always registered. Route tests (verify, batch-verify, user-lookup) construct Fastify manually: create instance, register sensible, correlation-id, logging, a fake firebase plugin (fp with name 'firebase' that decorates firebaseAuth with mock verifyIdToken/getUser), then register the route under test. Mock functions are vi.fn() on the decorated object. This pattern is proven in prior work on verify.spec.ts.


## Consequences

- Route tests are isolated — each test controls exactly what firebase methods return
- More setup boilerplate per test file — mitigate with shared test helper (tests/helpers/build-test-app.ts)
- No dependency on buildApp() internals for route tests
