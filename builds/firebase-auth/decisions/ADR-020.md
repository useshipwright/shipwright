# ADR-020: Operators Must Use Short-Lived Credentials (gcloud auth)

## Context

The threat model identifies "Privilege Escalation via Operator Over-Provisioning" as a medium-likelihood, high-impact threat. The operator running `deploy` requires powerful IAM roles including `roles/resourcemanager.projectIamAdmin` and `roles/iam.serviceAccountUser`. A compromised operator credential could escalate privileges across the entire GCP project.

Threat model mitigation (1) specifies: "use short-lived credentials (gcloud auth)". The RBAC matrix notes: "No programmatic API key — uses OAuth user credentials."

There are two ways an operator can authenticate to GCP:

1. **Exported service account key** (long-lived JSON file): Never expires unless manually rotated. If exfiltrated, grants persistent access with no revocation mechanism short of deleting the SA key.

2. **`gcloud auth login`** (short-lived OAuth token): Generates an access token that expires in ~1 hour. Backed by a refresh token that can be revoked centrally via Google Workspace or Cloud Identity. Requires interactive browser-based authentication (supports MFA).

## Decision

Operators MUST authenticate via `gcloud auth login` (interactive OAuth flow) rather than exported service account JSON keys. This is an operational policy enforced through documentation and deploy-time checks — not a code-level constraint.

### Policy requirements

1. **Interactive authentication**: Operators run `gcloud auth login` before `deploy` or `scripts/deploy.sh`. This produces a short-lived OAuth access token (~1 hour).

2. **No exported SA keys for operators**: Operators MUST NOT use `gcloud auth activate-service-account --key-file=...` with an exported JSON key. Exported keys are long-lived and cannot be centrally revoked without deleting the key from IAM.

3. **CI/CD exception**: Automated pipelines (GitHub Actions, Cloud Build triggers) may use Workload Identity Federation or short-lived OIDC tokens from the CI provider. These are acceptable because they are scoped, time-limited, and do not involve exportable key material.

4. **Deploy script check**: `scripts/deploy.sh` verifies `gcloud auth print-access-token` succeeds before proceeding. It also checks the active credential type and warns if a service account key is detected.

### How `gcloud auth login` provides short-lived credentials

- `gcloud auth login` opens a browser-based OAuth consent flow
- On success, gcloud stores a refresh token locally (~/.config/gcloud/)
- Each API call exchanges the refresh token for an access token (~1 hour TTL)
- The refresh token can be revoked centrally by a Workspace/Cloud Identity admin
- If MFA is enforced on the Google account, every new login requires MFA

### Why not exported SA keys?

| Property | `gcloud auth login` | Exported SA key |
|----------|---------------------|-----------------|
| Token lifetime | ~1 hour (auto-refreshed) | Never expires |
| Central revocation | Yes (Workspace admin) | Only by deleting key |
| MFA support | Yes | No |
| Audit trail | Tied to user identity | Generic SA identity |
| Exfiltration risk | Refresh token on local disk | JSON key file portable |
| CI/CD compatible | No (interactive) | Yes (but use WIF instead) |

## Consequences

- Operators must have Google accounts with sufficient IAM roles (not service accounts)
- Each operator action is attributable to a named user in Cloud Audit Logs
- Compromised credentials can be revoked centrally without rotating infrastructure
- CI/CD pipelines need Workload Identity Federation or equivalent (not exported keys)
- The deploy script provides a runtime check for credential type, reducing accidental use of SA keys
