# ADR-023: IAM Permission Diff Approval Step in Deployment Pipeline

## Context

The threat model identifies "Privilege Escalation via Operator Over-Provisioning" as a medium-likelihood, high-impact threat. Mitigation (4) specifies: "permission diff approval step provides audit trail."

The RBAC matrix lists `approve_permission_diffs` as a required operator permission: "Interactive approval required before IAM changes are applied."

The current deployment pipeline (`scripts/deploy.sh`) performs IAM operations across multiple phases (per-secret binding, custom role creation, Artifact Registry IAM audit, service account setup) but has no single gate where the operator reviews the cumulative IAM changes before they take effect. An operator running `deploy.sh` may not realize the full scope of IAM modifications until after they have been applied.

Without an approval step:
- An attacker who can trigger deploys (e.g., via compromised CI/CD) can silently modify IAM bindings.
- Operators cannot audit what changed during a deployment without reading GCP audit logs after the fact.
- There is no pre-apply review that could catch misconfigured or escalated permissions.

## Decision

Add Phase 0g to `deploy.sh` that computes the IAM permission diff between the desired state (from `deploy configuration` and `iam/custom-roles.yaml`) and the current GCP project state, then requires explicit operator approval before proceeding.

### Behaviour

1. **Compute desired state**: Read service account roles, custom role permissions, and Cloud Build SA bindings from the deploy configuration.
2. **Fetch current state**: Query the live GCP project for existing IAM bindings on the relevant service account and custom role definitions.
3. **Display diff**: Show a colour-coded summary of what will be added, removed, or unchanged.
4. **Require approval**: Prompt the operator interactively (`[y/N]`). Default is deny — the operator must explicitly type `y`.
5. **Write audit log**: On approval, append a timestamped entry to `logs/iam-audit.log` recording the operator identity, what was approved, and the deployment context.
6. **Support CI/CD**: `AUTO_APPROVE_IAM=true` skips the interactive prompt but still computes the diff and writes the audit log. The audit entry is marked `auto-approved` for traceability.
7. **Support skip**: `SKIP_IAM_DIFF=true` skips the entire phase with a warning. This is for environments where IAM is managed externally (e.g., Terraform).

### Audit log format

Each deployment appends a JSON-lines entry to `logs/iam-audit.log`:

```json
{
  "timestamp": "2026-02-19T12:00:00Z",
  "operator": "user@example.com",
  "project": "my-gcp-project",
  "service": "firebase-auth",
  "approval": "interactive",
  "changes": {
    "service_account_bindings": ["roles/secretmanager.secretAccessor (per-secret)"],
    "custom_roles": ["firebaseAuthPackOperator", "firebaseAuthPackBuilder"],
    "cloud_build_sa_bindings": ["firebaseAuthPackBuilder"]
  }
}
```

### Ordering

Phase 0g runs **before** any IAM-modifying phases (0b, 0d, 0f) and **before** Phase 1 (build). This ensures the operator approves the full scope of IAM changes before any of them are applied.

## Consequences

- Every deployment has a pre-apply IAM review — operators know exactly what permissions are being modified
- Audit log provides a persistent, searchable record of who approved what and when
- CI/CD pipelines must explicitly set `AUTO_APPROVE_IAM=true` — accidental auto-approval is prevented
- The audit log file (`logs/iam-audit.log`) should be committed to version control or shipped to a central logging system for compliance
- Interactive mode is the default — operators cannot accidentally skip the review
- This adds ~2-5 seconds to interactive deployments and is a no-op for `SKIP_IAM_DIFF=true`
