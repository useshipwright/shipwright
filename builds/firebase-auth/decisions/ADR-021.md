# ADR-021: Require MFA on Operator Accounts via Cloud Identity Policy

## Context

The threat model identifies "Privilege Escalation via Operator Over-Provisioning" as a medium-likelihood, high-impact threat. Mitigation (2) specifies: "require MFA on operator accounts." ADR-020 already requires operators to use `gcloud auth login` (short-lived OAuth credentials), which supports MFA — but does not guarantee MFA is enforced.

MFA enforcement is a GCP organization-level policy managed through Cloud Identity or Google Workspace. It cannot be enforced in application source code or at the individual project level. However, the deploy script can detect whether the active operator credential is a user account (which supports MFA) versus a service account (which does not).

### How MFA enforcement works in GCP

1. **Cloud Identity / Google Workspace admin** enables the "Enforce MFA" policy for the organizational unit containing operator accounts.
2. When an operator runs `gcloud auth login`, the OAuth flow prompts for MFA if the admin policy requires it.
3. If MFA is not enforced, operators can still authenticate with password-only — the deploy script cannot distinguish MFA-authenticated sessions from non-MFA sessions.

### Why application code cannot enforce MFA

- `gcloud auth print-access-token` returns an opaque OAuth access token — there is no claim or header indicating whether MFA was used during authentication.
- The GCP IAM API does not expose session MFA state to downstream consumers.
- MFA enforcement is an identity provider concern, not an application concern.

## Decision

MFA enforcement for operator accounts is an organizational policy requirement, documented here and verified through operational controls:

### Policy requirements

1. **Cloud Identity MFA policy**: All Google accounts used for `deploy` or `scripts/deploy.sh` MUST have MFA enforced via Cloud Identity or Google Workspace admin console. This is an organizational responsibility, not an application-level control.

2. **Deploy script detection**: `scripts/deploy.sh` already verifies the active credential is a user account (ADR-020). Service accounts do not support MFA. The deploy script warns when a service account is detected, which implicitly covers the MFA gap.

3. **Documentation**: The usage guide documents MFA as a prerequisite for operator accounts, directing administrators to the Cloud Identity admin console for enforcement.

### How to enable MFA enforcement

1. Navigate to **Google Admin Console** > **Security** > **2-Step Verification**
2. Select the organizational unit containing operator accounts
3. Set enforcement to **On** with an enrollment deadline
4. Recommended: allow only **security keys** and **Google prompts** (disable SMS/voice as backup)

### Verification

There is no programmatic way to verify MFA enforcement from the deploy script. Instead:

- **Cloud Audit Logs** record login events with authentication method details
- Administrators can audit MFA enrollment status in the Cloud Identity admin console
- The deploy script's ADR-020 credential type check ensures operators use `gcloud auth login`, which is the only authentication flow that supports MFA

## Consequences

- MFA enforcement depends on organizational Cloud Identity policy — not application code
- Operators who bypass `gcloud auth login` (e.g., using exported SA keys) also bypass MFA; ADR-020's deploy script check mitigates this
- Organizations without Cloud Identity or Google Workspace cannot enforce MFA via policy — they must rely on operator discipline
- The deploy script provides a "best-effort" check by ensuring user credentials are used (ADR-020), but cannot verify MFA was actually performed during the session
