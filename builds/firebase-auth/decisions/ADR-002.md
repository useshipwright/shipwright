# ADR-002: In-memory per-instance rate limiting (no Redis)

## Context

PRD requires per-API-key rate limiting with three classes. Cloud Run autoscales across instances.
ASSUMP-003 explicitly states rate limiting is per-instance and effective limits scale linearly
with instance count. Adding Redis would introduce operational complexity and a new failure mode.


## Decision

Implement rate limiting as an in-memory sliding window counter (Map<compositeKey, bucket>) inside
a Fastify preHandler hook. No external dependencies. Each Cloud Run instance tracks its own counters
independently. Counters reset when instances recycle.


## Consequences

+ Zero additional infrastructure — no Redis to provision, monitor, or pay for.
+ No network latency added to every request for rate limit checks.
+ Graceful degradation — instance restart clears counters (brief burst allowed, not harmful).
- Effective rate limits are N × configured limit where N = instance count. At max-instances=1, exact.
  At higher scale, limits are looser than configured. PRD accepts this (ASSUMP-003).
- No rate limit state sharing across instances. Acceptable for internal service-to-service traffic.
