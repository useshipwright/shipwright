# ADR-002: Batch-verify returns per-token error categories while single verify returns generic 401

## Context

REQ-008 requires generic 401 for verify failures. REQ-002 requires per-token results with reason for batch-verify. Q-003 flags this inconsistency. Both endpoints are internal-only.


## Decision

Single verify: binary pass/fail. 200 with claims or 401 with generic message. This is the simplest contract for callers doing pass/fail token gates. Batch-verify: 200 always (batch succeeded), with per-token valid/invalid status. Invalid tokens include a coarse error category (expired, invalid, malformed) — NOT the raw Firebase error code. This lets callers distinguish actionable cases (tell client to refresh vs re-authenticate) without leaking cryptographic details.


## Consequences

- Callers can use single verify for simple gates and batch-verify for bulk processing with per-token handling
- Error categories are coarse (3 values) — maps multiple Firebase error codes to generic buckets
- expired = auth/id-token-expired; malformed = structural validation failure; invalid = everything else
