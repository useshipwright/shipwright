# ADR-019: Support Workload Identity Federation via Application Default Credentials

## Context

The threat model identifies "Service Account Key Exfiltration" as a high-impact risk. Mitigation (3) recommends: "consider Workload Identity Federation instead of SA key for v1.1".

The v1 implementation uses a long-lived Firebase service account JSON key (`FIREBASE_SERVICE_ACCOUNT_JSON`) stored in Secret Manager and mounted as an environment variable at Cloud Run revision creation time. This key:

- Contains a private key that never expires unless manually rotated
- Exists in container memory for the process lifetime
- Grants full Firebase Auth admin access if exfiltrated
- Requires Secret Manager infrastructure (secret creation, per-secret IAM binding, audit logging)

Workload Identity Federation (WIF) eliminates long-lived key material entirely. On Cloud Run, the service runs with a GCP service account identity provided by the metadata server. The Firebase Admin SDK's `applicationDefault()` credential obtains short-lived tokens from the metadata server — no private key is stored anywhere.

## Decision

The adapter (`src/adapters/firebase-admin.ts`) supports two credential modes:

1. **SA JSON key** (v1 default): `initFirebase(credentialJson)` — uses `cert()` with the parsed service account JSON. Activated when `FIREBASE_SERVICE_ACCOUNT_JSON` is set.

2. **Application Default Credentials** (WIF): `initFirebaseWithADC()` — uses `applicationDefault()`. Activated when `FIREBASE_USE_ADC=true` and `FIREBASE_SERVICE_ACCOUNT_JSON` is not set.

The firebase plugin (`src/plugins/firebase.ts`) checks environment variables in this priority order:

1. `FIREBASE_SERVICE_ACCOUNT_JSON` → SA JSON key (takes precedence if both are set)
2. `FIREBASE_USE_ADC=true` → Application Default Credentials
3. Neither → fail-fast (container won't start)

If both are set, the plugin logs a warning and uses the SA JSON key.

### Deploy script changes

`scripts/deploy.sh` accepts `FIREBASE_USE_ADC=true` to deploy without Secret Manager secrets:

- **SA JSON key mode** (default): `--set-secrets="FIREBASE_SERVICE_ACCOUNT_JSON=firebase-auth-sa-key:latest"`
- **WIF mode**: `--set-env-vars="FIREBASE_USE_ADC=true"` (no `--set-secrets`)

### WIF prerequisites (operator responsibility)

To use WIF mode, the operator must:

1. Ensure the Cloud Run service account has Firebase Auth permissions (e.g., `roles/firebaseauth.admin` or a custom role supporting `verifyIdToken()` and `getUser()`)
2. The Cloud Run service account must be a GCP service account (not a user account)
3. No `FIREBASE_AUTH_EMULATOR_HOST` env var should be set (WIF + emulator mode is undefined behavior)

### Why not remove SA JSON key support?

- SA JSON key works in all environments (local dev, CI, Cloud Run, other cloud providers)
- WIF only works in GCP environments with metadata server access
- Local development requires either SA JSON key or `gcloud auth application-default login`
- Gradual migration: existing deployments can switch to WIF without code changes

## Consequences

- Eliminates long-lived key material when WIF is used — no private key in Secret Manager or container memory
- Reduces Secret Manager dependency (no secret creation, IAM binding, or audit logging needed for the Firebase credential)
- Credential rotation is automatic — metadata server tokens are short-lived (~1 hour) and auto-refreshed
- SA JSON key remains supported for backward compatibility and non-GCP environments
- Operators choosing WIF must ensure proper IAM roles on the Cloud Run service account
- The `FIREBASE_AUTH_EMULATOR_HOST` security concern (threat model) applies equally to both modes
