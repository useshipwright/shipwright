# ADR-010: Constant-time operations scope

## Context

REQ-011 requires constant-time security operations. Q-014 asks what this means given verifyIdToken() itself is not constant-time. ASSUMP-008 clarifies this refers to string comparisons and error-path timing normalisation.


## Decision

Apply constant-time principles in two areas: (1) Use timing-safe comparison (crypto.timingSafeEqual) for any direct string comparisons on security-sensitive values (if any arise — currently verifyIdToken handles this internally). (2) Normalise error response timing — add a minimum response delay on the verify endpoint so that invalid tokens do not return faster than valid ones (prevents timing oracle). In practice, the Firebase SDK's network calls dominate latency, making timing attacks impractical, but the normalisation is a defence-in-depth measure.


## Consequences

- Defence-in-depth — unlikely to be exploitable but demonstrates security posture
- Minimum response delay adds ~50-100ms worst case to fast-failing requests
- Implementation is simple — setTimeout or await a timer before sending error responses
