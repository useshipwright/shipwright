# ADR-015: Image Digest Pinning in Cloud Run Deploy

## Context

The threat model identifies "Container Image Tampering" as a risk: if Artifact Registry permissions are too broad, an attacker could replace a tagged container image with a malicious one. Tags (including `latest`) are mutable pointers — pushing a new image with the same tag silently replaces the previous one. A Cloud Run service deployed with `--image=repo/image:tag` could be updated to run a tampered image without any change to the deploy command.

**Threat model mitigation (1):** "use image digest pinning in Cloud Run deploy."

## Decision

The deploy script (`scripts/deploy.sh`) resolves the immutable sha256 digest of the just-built image immediately after `gcloud builds submit`, then deploys to Cloud Run using the `image@sha256:<digest>` format instead of `image:tag`.

### Implementation

1. **Build phase** pushes the image with a tag (for human readability and registry browsing).
2. **Digest resolution phase** calls `gcloud artifacts docker images describe` to retrieve the image's sha256 digest.
3. **Deploy phase** uses `IMAGE_REPO@sha256:<digest>` as the `--image` argument to `gcloud run deploy`.
4. The script **fails fast** if the digest cannot be resolved (empty result or command failure).

### Why not tag-based deploy?

| Property | Tag (`:latest`) | Digest (`@sha256:...`) |
|----------|-----------------|------------------------|
| Mutable | Yes — can be overwritten | No — content-addressed |
| Auditability | Tag points to unknown content | Digest is cryptographic proof |
| Replay safety | Deploying same tag may deploy different code | Same digest always means same image |
| Rollback | Ambiguous — tag may have moved | Deterministic — exact image bytes |

## Consequences

- Every Cloud Run deploy is cryptographically tied to the exact image that was just built
- Artifact Registry tag tampering cannot affect running deployments (revision is pinned to digest)
- Cloud Run revision history shows exact digests for auditing
- Tags are still pushed for human convenience (browsing Artifact Registry console) but are not trusted for deployment
- The deploy script requires an additional `gcloud artifacts docker images describe` call (adds ~2-3 seconds)
