# ADR-016: Binary Authorization Deferred to v1.1

## Context

The threat model identifies "Container Image Tampering" as a risk: if Artifact Registry permissions are too broad, an attacker could replace a tagged container image with a malicious one. Mitigation item (4) recommends: "consider Binary Authorization for deploy-time verification."

Binary Authorization (BinAuthz) is a GCP service that enforces deploy-time security policies on Cloud Run and GKE. It can require that container images are signed by trusted attestors before they are allowed to run, providing a cryptographic chain of trust from build to deploy.

### Current mitigations already in place

v1 already implements three of the four threat model mitigations for container image tampering:

1. **Image digest pinning** (ADR-015) — deploy.sh resolves the sha256 digest after build and deploys using `image@sha256:<digest>` instead of a mutable tag.
2. **Artifact Registry writer restriction** (T-048) — `roles/artifactregistry.writer` is scoped to the Cloud Build SA only, verified by an IAM audit step in deploy.sh.
3. **Artifact Analysis vulnerability scanning** (T-049) — Container Analysis and Container Scanning APIs are enabled; deploy.sh checks for CRITICAL CVEs before deploying.

### Why not v1?

Binary Authorization adds significant operational complexity:

- Requires creating and managing attestor key pairs (Cloud KMS or PGP)
- Requires a custom Cloud Build step to create attestations after successful builds
- Requires configuring a BinAuthz policy on the Cloud Run service
- Requires the operator to hold additional IAM roles (`roles/binaryauthorization.policyEditor`, `roles/binaryauthorization.attestorsEditor`)
- Requires enabling the Binary Authorization API (`binaryauthorization.googleapis.com`)
- Breakglass procedures must be documented for emergency deploys that bypass policy

For a v1 PoC with internal-only ingress and digest-pinned deploys, the existing three mitigations provide sufficient protection against image tampering. The residual risk is low: an attacker would need to compromise both the Cloud Build SA credentials AND the Artifact Registry IAM policy to inject a tampered image, and even then the digest-pinned deploy would only pick up an image built in the same pipeline run.

## Decision

Defer Binary Authorization to v1.1. The three existing mitigations (digest pinning, AR writer restriction, vulnerability scanning) are sufficient for the v1 threat profile. Binary Authorization should be reconsidered if:

- The service moves to external ingress (increased attack surface)
- Multiple teams gain write access to the Artifact Registry repository
- Compliance requirements mandate attestation-based deploy policies
- A security incident involving image tampering occurs

### v1.1 implementation sketch (for future reference)

1. Enable `binaryauthorization.googleapis.com` API
2. Create a Cloud KMS key for attestation signing
3. Create a BinAuthz attestor linked to the KMS key
4. Add a Cloud Build step that creates an attestation after successful image push
5. Configure a BinAuthz policy on the Cloud Run service requiring the attestation
6. Add breakglass annotation support in deploy.sh for emergency deploys
7. Add operator roles: `roles/binaryauthorization.policyEditor`, `roles/binaryauthorization.attestorsEditor`

## Consequences

- v1 does not enforce attestation-based deploy policies — this is an accepted risk documented in the usage guide
- The existing three-layer mitigation (digest pinning + AR IAM audit + vulnerability scanning) provides defense-in-depth for v1
- deploy.sh includes a placeholder comment indicating where BinAuthz verification would be inserted
- When v1.1 adds BinAuthz, the deploy script already has the correct phase structure (build → scan → verify → deploy) to insert an attestation step
