# ADR-003: API key ID derived from SHA-256 hash prefix

## Context

Q-002 asks how api-key-id is derived for audit logs. API_KEYS is a comma-separated list of opaque
keys. Options: positional index (fragile if list changes), named keys (format change), hash prefix.


## Decision

Derive key ID as sha256(key).slice(0, 8). Computed once at startup when parsing API_KEYS into a
Map<keyId, keyBuffer>. The 8-char hex prefix provides collision resistance (4 billion possibilities)
sufficient for audit correlation while being non-reversible. Stored in config as a lookup map.


## Consequences

+ Deterministic — same key always produces same ID across deployments and instances.
+ Non-reversible — key cannot be recovered from ID (important for audit log security).
+ No format change to API_KEYS env var — operators don't need to learn a naming convention.
- 8 hex chars has theoretical collision risk at ~65k keys (birthday bound). With typical deployments
  of 1–10 API keys, collision probability is negligible.
