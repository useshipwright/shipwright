# ADR-003: Firebase Admin SDK handles JWKS caching — no custom cache

## Context

REQ-012 requires JWKS caching with auto-refresh. Domain research (ASSUMP-005) confirms the SDK caches public keys in-memory with Cache-Control TTL (~6 hours). Cloud Run instances are long-lived enough for this cache to be effective.


## Decision

Rely on Firebase Admin SDK's built-in UrlKeyFetcher for JWKS caching. No custom caching layer. Log a message when the SDK fetches keys (observable via startup timing) but do not wrap or intercept the SDK's key management.


## Consequences

- REQ-012 is satisfied by SDK defaults — no custom code to maintain
- Cold starts (new Cloud Run instances) always fetch keys on first verifyIdToken()
- Key rotation handled by Cache-Control expiry — SDK fetches fresh keys when TTL expires
- If a kid mismatch occurs mid-TTL (rare rotation edge case), the request fails and the next request refetches
